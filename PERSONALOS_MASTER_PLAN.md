# PersonalOS Master Plan

## Purpose

This document is the canonical map of PersonalOS.

PersonalOS is not primarily a task manager. It is a human-centered digital refuge designed to help a person recover clarity, preserve balance, and continue their path one step at a time.

## Ecosystem alignment

PersonalOS belongs to the Balance Ecosystem.

- PersonalOS: flourishing of the person.
- ATLAS: flourishing of knowledge.
- CYGNUS: flourishing of the organization.

Shared principle:

> Technology should not occupy the center of human life. Technology should create the conditions for people, knowledge, and organizations to flourish without losing humanity.

## Product promise

PersonalOS does not exist to capture attention.

It exists to help the person return to the world a little clearer, calmer, and more able to live outside the system.

## Foundational principles

1. The person is always more important than the system.
2. Reduce before adding.
3. Never increase guilt, shame, or anxiety.
4. Every path can be resumed.
5. Attention is sacred.
6. Autonomy always wins.
7. Silence also communicates.
8. Everything should feel reversible.
9. Memory exists to care, not to watch.
10. The success of PersonalOS is measured outside the system.

## First Experience

The first experience must ask only what is necessary to create the first Refugio.

Flow:

1. Welcome.
2. Ask who the Refugio is for using a broad life profile.
3. Ask how the person wants to be called.
4. Ask preferred companion style.
5. Create one initial person only.
6. Create personalized Refugio.
7. Enter through Mellon.

The initial installer must not create a hardcoded family structure. Additional people grow later from Mi Jardin.

## Companion model

Companions are archetypes of accompaniment.

| Companion | Archetype | Tone |
|---|---|---|
| Aurora | Serene presence | Calm, soft, contemplative |
| Samwise | Companion of the road | Close, steady, encouraging |

Companion selection is affinity-based, not gender-based.

## Resource Framework

A Paso may require a Resource.

If the Resource is not configured, PersonalOS asks one small question at the right moment.

Example:

1. Paso: Open Classroom.
2. PersonalOS checks if Classroom is configured.
3. If not, it asks how the person prefers to open it.
4. The answer unlocks a capability.
5. PersonalOS remembers and continues.

## Progressive Familiarity

Rules:

1. Never ask for information before it is needed.
2. Every question must unlock a capability.
3. Every question must give something back.
4. The system must not overwhelm at first contact.
5. Trust grows through repeated respectful interactions.

## Roadmap

### Phase 0 — Vision

Status: complete.

Outcome: philosophical and experiential identity established.

### Phase 1 — Experience Architecture

Define Progressive Familiarity, Companion Framework, Resource Framework, First Experience, Experience Language, and Refugio architecture.

### Phase 2 — Notion v0.2

Implement First Experience, personalized first person, companion selection, Resource setup for Classroom, cleaner page titles, safe icon handling, and no hardcoded family setup.

### Phase 3 — Aurora Core v0.1

Implement platform-neutral models and deterministic engines.

### Phase 4 — Mobile Experience

Use Aurora Core and the same experience language outside Notion.

### Phase 5 — Aurora Runtime

Introduce deeper adaptation, reflection, wisdom, and meaningful memory.

## Final compass

PersonalOS must always answer:

> Does this help the person return to life with more clarity, balance, dignity, or meaning?

If not, it does not belong in the core experience yet.
